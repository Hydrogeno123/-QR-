# advanced_schemes.py 代码详解

## 一、整体思路

本文件实现了三种基于视觉密码学（VC）的 QR 认证方案。三种方案的共同目标是：

1. 将一个**秘密 QR**（包含认证令牌）拆成两个份额（share）。
2. 两个份额各自看起来像无意义的噪声或混合图像。
3. 将两个份额叠加（OR 操作）后，可以恢复出可扫码的秘密 QR。

三种方案的复杂度逐级递增：Scheme I → Scheme II → Scheme III，隐蔽性也逐级增强。

---

## 二、数据表示约定

理解代码之前，必须先理解数据表示：

| 概念 | 代码中的值 | 像素中的值 |
|------|-----------|-----------|
| 黑模块 | `1` | `0`（纯黑像素） |
| 白模块 | `0` | `255`（纯白像素） |

**叠加规则**：`stack(a, b) = a OR b`（黑色主导）

- 只要任一份额在该位置是黑（1），叠加结果就是黑。
- 只有两个份额在该位置都是白（0），叠加结果才是白。

这个规则是所有算法构造的基础——它决定了哪些 `(q3, q4)` 的组合是"合法的"。

## 三、公共工具函数

### 3.1 `_generate_qr_modules(data)` — 生成 QR 模块矩阵

将文本数据编码为 QR 码，提取其模块矩阵，返回一个 `0/1` 的 `numpy` 数组。

关键参数：`version=3`（固定 QR 版本）、`ERROR_CORRECT_H`（最高纠错等级）、`border=0`（无边框）。

> 为什么固定版本？因为后续的 XOR、OR 运算要求所有矩阵尺寸一致。version 3 → 29×29 模块。

### 3.2 `_modules_to_image(modules, scale=10)` — 矩阵转图片

将 `0/1` 模块矩阵映射为灰度图：
- `1`（黑模块）→ 像素值 `0`
- `0`（白模块）→ 像素值 `255`

`scale=10` 表示每个模块放大 10 倍（29×29 → 290×290），否则图像太小，扫码器无法识别。

### 3.3 `_save_scheme(out_dir, secret, q3, q4)` — 保存结果

保存 5 张标准输出图：原始秘密 QR、份额 1、份额 2、叠加图、恢复图。

## 四、核心算法：非扩展 (2,2)-VCS 分影

### `_generate_non_expansion_shadows(secret)`

将秘密模块矩阵拆成两个份额：

```
对于每个位置 (i, j)：
  如果 secret[i,j] == 0（白）：
    share1[i,j] = 0,  share2[i,j] = 0
    → 叠加：0 OR 0 = 0（白）✓

  如果 secret[i,j] == 1（黑）：
    随机选择：(1, 0) 或 (0, 1) 分配给两个份额
    → 叠加：1 OR 0 = 1（黑）✓
```

**为什么单个份额看不出秘密？** 因为白色位置 `(0,0)` 是全白，黑色位置随机分配，单张份额看起来只是随机噪声。

---

## 五、核心算法：约束融合求解器

### `_fuse_with_targets_under_secret_constraint(t1, t2, secret)`

**这是整个文件中最关键的函数**，三个方案都依赖它。

**问题描述**：给定目标图像 `t1`、`t2` 和秘密矩阵 `secret`，找到 `(q3, q4)` 使得：
1. **硬约束**：`q3 OR q4 == secret`（叠加必须恢复秘密）
2. **软目标**：`(q3, q4)` 尽量接近 `(t1, t2)`（看起来像目标图像）

**求解过程**（逐模块贪心）：

```
对于每个位置 (i, j)：

  如果 secret[i,j] == 0（白）：
    唯一合法组合：(0, 0)，因为 0 OR 0 = 0
    → 强制设为 (0, 0)
    → 即使目标 (t1, t2) 在此处是 (1, 0)，也只能接受差异

  如果 secret[i,j] == 1（黑）：
    合法组合有三种：(1,0)、(0,1)、(1,1)
    → 分别计算与目标 (t1, t2) 的距离：
      - (1,0) 的距离 = (1-t1)² + (0-t2)²
      - (0,1) 的距离 = (0-t1)² + (1-t2)²
      - (1,1) 的距离 = (1-t1)² + (1-t2)²
    → 选距离最小的那个
```

**举例**：假设 `secret=1`，目标 `(t1,t2) = (1,0)`：

| 候选 | 距离 | 结果 |
|------|------|------|
| (1, 0) | (1-1)² + (0-0)² = **0** | 最优 |
| (0, 1) | (0-1)² + (1-0)² = 2 | |
| (1, 1) | (1-1)² + (1-0)² = 1 | |

选 (1, 0)，完美匹配目标。

**向量化实现**：代码没有用 Python 循环逐像素处理，而是对所有 `secret==1` 的位置一次性计算三个候选的距离，用 `argmin` 选出最优。这比循环快几个数量级。

## 六、背景生成函数

### `_binary_background(shape, seed)` — 随机二值背景

用固定种子生成随机的 `0/1` 矩阵。用于 Scheme II。

### `_color_background(shape, seed)` — 彩色渐变背景

用正弦函数在 R/G/B 三个通道上生成平滑渐变图案，返回 `(H, W, 3)` 的彩色数组。用于 Scheme III。

三个通道分别基于不同的空间坐标和随机相位，使背景看起来色彩丰富。

## 七、显著性与功能区掩码

### `_grayscale_saliency(color_bg)` — 梯度显著性

计算彩色图像的"视觉重要程度"：

1. 将彩色图转灰度：`gray = 0.299R + 0.587G + 0.114B`
2. 用 `np.gradient` 计算水平方向 `gx` 和垂直方向 `gy` 的梯度
3. 梯度幅值 = `sqrt(gx² + gy²)`
4. 归一化到 `[0, 1]`

**直觉**：边缘和纹理区域梯度大（显著性高），平坦区域梯度小（显著性低）。修改低显著性区域最不容易被人眼察觉。

### `_functional_mask_like_qr(shape)` — QR 功能区掩码

返回布尔矩阵，`True` 表示"受保护，禁止修改"。

对于 29×29 的 QR（version 3），受保护的区域包括：

```
位置                  | 保护范围              | 含义
----------------------|----------------------|------
左上定位图案          | [0:8, 0:8]          | 8×8（含分隔线）
右上定位图案          | [0:8, 21:29]        | 8×8
左下定位图案          | [21:29, 0:8]        | 8×8
水平时序图案          | 第 6 行，第 8~20 列  | 交替黑白条纹
垂直时序图案          | 第 6 列，第 8~20 行  | 交替黑白条纹
对齐图案              | [20:25, 20:25]      | 5×5（中心在 22,22）
```

> 为什么要保护？这些区域是扫码器识别 QR 的关键结构。如果被修改，QR 可能无法被识别。

1. 叠加：`stacked = share1 | share2`

## 九、Scheme I：BVC（载体 QR 融合）

**思路**：将 VCS 分影与载体 QR 进行融合，使份额看起来像"混合了载体内容的噪声"。

### 流程

```
1. 生成秘密 QR：secret = _generate_qr_modules("AUTH-2024-VC-DEMO")
   → 29×29 的 0/1 矩阵

2. 生成载体 QR：carrier = _generate_qr_modules("https://example.com/a")
   → 同样 29×29

3. VCS 分影：s1, s2 = _generate_non_expansion_shadows(secret)
   → 两个份额，叠加后等于 secret

4. 构造融合目标：
   t1 = carrier XOR s1
   t2 = carrier XOR s2
   → 用 XOR 将载体和份额混合，作为"期望的外观"

5. 约束融合：q3, q4 = _fuse_with_targets_under_secret_constraint(t1, t2, secret)
   → 找到最接近目标且满足 q3 OR q4 = secret 的组合

6. 保存并验证
```

### 直觉

XOR 操作将载体 QR 的模块与 VCS 份额"搅合"在一起。求解器在满足叠加约束的前提下，尽量让结果接近这个搅合后的目标。最终的份额既不是纯噪声，也不是纯载体，而是两者的混合。

## 十、Scheme II：One-Meaningful（两级融合链）

**思路**：在 Scheme I 的基础上增加一层"背景融合"，使份额看起来更自然。

### 流程

```
1. 生成秘密 QR 和载体 QR（同 Scheme I）

2. VCS 分影：s1, s2

3. 生成两张随机二值背景：bg1（seed=42），bg2（seed=99）

4. 第一级融合（阴影 → 背景）：
   fused_bg1 = bg1 XOR s1
   fused_bg2 = bg2 XOR s2
   → 将阴影信息"藏"进背景

5. 第二级融合（背景 → 载体）：
   t1 = carrier XOR fused_bg1
   t2 = carrier XOR fused_bg2
   → 再将"含阴影的背景"与载体混合

6. 约束融合：q3, q4 = _fuse_with_targets_under_secret_constraint(t1, t2, secret)
```

### 与 Scheme I 的区别

Scheme I 只有一级融合（阴影 ↔ 载体），Scheme II 有两级（阴影 → 背景 → 载体）。两级链使得最终份额更像"背景 + 载体"的混合，视觉上更自然，隐蔽性更高。

### XOR 链的数学性质

```
fused_bg = bg XOR s      （阴影被编码进背景）
t = carrier XOR fused_bg （背景被编码进载体）

展开：t = carrier XOR bg XOR s
```

三次 XOR 形成一条链。最终的份额 `q3/q4` 是对 `t` 的近似，叠加时通过 OR 约束恢复 `s`。

## 十一、Scheme III：Two-Meaningful（ROI + 显著性 + 预算）

**这是最复杂的方案**，引入了三个新概念：ROI（感兴趣区域）、显著性、纠错预算。

### 流程

```
1. 生成秘密 QR 和两个载体 QR（carrier_a, carrier_b）

2. VCS 分影：s1, s2

3. 生成彩色渐变背景 → 灰度化 → 二值化 → bg_bin
   （彩色图仅用于计算显著性，二值图用于实际融合）

4. 定义 ROI = 中心 15×15 区域 [7:22, 7:22]
   （避开三个角落的定位图案）

5. 将阴影注入 ROI：
   bg1[ROI] = s1[ROI]     ← ROI 内是阴影
   bg2[ROI] = s2[ROI]
   bg1[非ROI] = bg_bin     ← ROI 外是背景
   bg2[非ROI] = bg_bin

6. 计算显著性图（哪些位置改了最不显眼）

7. 构造功能区掩码（哪些位置绝对不能动）

8. 构造融合目标：
   t1 = carrier_a XOR bg1
   t2 = carrier_b XOR bg2

9. 【关键】全局约束融合：q3, q4 = _fuse_with_targets_under_secret_constraint(t1, t2, secret)
   → 保证 q3 OR q4 = secret 在所有位置成立

10. 预算统计：
    - 候选位置 = ROI 内 ∩ 非功能区
    - 按显著性排序（低在前）
    - 取前 30% 作为"预算内"位置
    - 统计这些位置上 q3/q4 与目标的偏差数
```

### 为什么 Scheme III 有两个载体？

Scheme I/II 只用一个载体，两个份额融合的目标都来自同一个载体。Scheme III 用两个不同的载体（`carrier_a` 和 `carrier_b`），使两个份额看起来各自像不同的 QR 码。这就是"Two-Meaningful"的含义——两个份额各自有意义。

### ROI 的作用

QR 码的角落是定位图案（扫码器必须识别），不能动。ROI 定义在中心区域，将阴影信息集中注入这里，避免破坏角落的功能区。

### 显著性的直觉

```
显著性高 = 边缘、纹理区域 = 人眼敏感 = 尽量不改
显著性低 = 平坦区域 = 人眼不敏感 = 可以改
```

代码将候选位置按显著性从低到高排序，优先使用"最不显眼"的位置。

### 纠错预算（30%）

QR 码的 H 级纠错可以恢复约 30% 的损坏模块。预算 = 总模块数 × 30% = 841 × 30% ≈ 252。

这个预算在代码中用于**统计**：在候选位置中，有多少处 q3/q4 与目标产生了偏差。只要偏差不超过纠错容量，扫码器就能恢复正确数据。

### Scheme III 的关键设计决策

在调试过程中发现，如果只在预算内的位置执行约束融合，其余位置保持目标值不变，那么叠加结果会有 ~45% 的模块与秘密不一致——远超 H 级纠错的 30% 容量，导致解码失败。

因此改为：**全局执行约束融合**，保证 `q3 OR q4 == secret` 在所有位置成立（完美恢复）。预算仅用于统计偏差量，衡量对载体 QR 可读性的影响。

## 十二、三种方案对比

| 特性 | Scheme I | Scheme II | Scheme III |
|------|----------|-----------|------------|
| 载体数 | 1 个 | 1 个 | 2 个 |
| 背景 | 无 | 二值随机 | 彩色渐变 |
| 融合层级 | 1 级 | 2 级 | 1 级 + ROI 注入 |
| 显著性引导 | 无 | 无 | 有 |
| 纠错预算 | 无 | 无 | 30%（统计用） |
| 份额外观 | 噪声 + 载体 | 噪声 + 背景 + 载体 | 背景 + 双载体 |
| 隐蔽性 | 低 | 中 | 高 |
| 叠加恢复 | 严格恢复 | 严格恢复 | 严格恢复 |
